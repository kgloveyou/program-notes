这三个文件在前端项目中有不同的用途和特点：

## 📋 文件对比

### **`.env`** - 本地环境变量文件
- **用途**：存储**实际的敏感信息**（API密钥、token等）
- **提交到Git**：❌ **不应该提交**（需要在 `.gitignore` 中）
- **何时使用**：本地开发时使用，包含真实的凭证
- **示例内容**：
  ```env
  VITE_CESIUM_ION_TOKEN=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...（真实token）
  VITE_API_KEY=sk_live_abc123xyz...（真实密钥）
  ```

### **`.env.local`** - 本地个人配置文件
- **用途**：存储**个人特定的配置**，优先级高于 `.env`
- **提交到Git**：❌ **不应该提交**（需要在 `.gitignore` 中）
- **何时使用**：当团队成员需要不同配置时（如不同的数据库连接、调试设置）
- **示例内容**：
  ```env
  VITE_API_URL=http://localhost:3000  # 个人本地后端地址
  VITE_DEBUG_MODE=true
  ```

### **`.env.example`** - 环境变量模板文件
- **用途**：提供**配置模板和说明**，告诉其他开发者需要哪些环境变量
- **提交到Git**：✅ **应该提交**（作为项目文档的一部分）
- **何时使用**：作为新开发者的配置参考
- **示例内容**：
  ```env
  # Cesium Ion Access Token
  # Get your token from: https://cesium.com/ion/tokens
  VITE_CESIUM_ION_TOKEN=your_cesium_ion_token_here
  
  # API Configuration
  VITE_API_KEY=your_api_key_here
  ```

## 🔄 加载优先级（Vite）

```
.env.local > .env > .env.example
```

如果同一个变量在多个文件中定义，`.env.local` 的值会覆盖 `.env`。

## 💡 最佳实践

在你的项目中：

1. **`.env`** - 包含真实的 Cesium Ion token（已添加到 `.gitignore` ✅）
2. **`.env.example`** - 包含配置模板，提交到Git让其他开发者参考（已创建 ✅）
3. **`.env.local`** - 如果需要个人特定配置可以创建，也要加入 `.gitignore`（已添加 ✅）

## 📝 新开发者使用流程

```bash
# 1. 克隆项目后
git clone <repository>

# 2. 复制模板文件
cp .env.example .env

# 3. 编辑 .env，填入自己的真实 token
# VITE_CESIUM_ION_TOKEN=<你的token>

# 4. 安装依赖并运行
pnpm install
pnpm dev
```

这样既保证了安全性（敏感信息不会泄露），又方便了团队协作（有清晰的配置指引）。🔒
